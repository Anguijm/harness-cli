# `.harness/` — local protocol

This directory holds the council reviewers, runner script, durable session state, and learnings KB. Generated by `harness init`; specialize per-repo by editing the persona scope sections and the security checklist.

## Layout

```
.harness/
├── README.md                 # this file
├── halt_instructions.md      # circuit-breaker docs
├── model-upgrade-audit.md    # checklist when swapping models
├── active_plan.md            # current work plan (template — fill in per session)
├── learnings.md              # append-only KEEP/IMPROVE/INSIGHT/COUNCIL log
├── session_state.json        # auto-updated by post-commit hook
├── yolo_log.jsonl            # append-only audit trail
├── last_council.md           # most recent council report (overwritten)
├── council/                  # persona files (one per reviewer)
├── scripts/
│   ├── council.py            # the runner
│   ├── install_hooks.sh      # one-time setup (sets core.hooksPath)
│   ├── requirements.txt      # python deps for council
│   └── security_checklist.md # injected into Security reviewer prompt
└── hooks/
    └── post-commit           # updates session_state + yolo_log
```

## One-time setup

```bash
# install the post-commit hook
bash .harness/scripts/install_hooks.sh

# install python deps for the local council
pip install -r .harness/scripts/requirements.txt

# set GEMINI_API_KEY in your shell (also needed as a repo secret for CI)
export GEMINI_API_KEY=...
```

## Local council usage

```bash
# review a plan file (must be tracked in git)
python3 .harness/scripts/council.py --plan .harness/active_plan.md

# review the working diff vs origin/main
python3 .harness/scripts/council.py --diff --base origin/main
```

The CI workflow (`.github/workflows/council.yml`) does the same thing automatically on every PR.

## Halt

To stop all council automation: `echo "reason" > .harness_halt` at repo root. See `halt_instructions.md`.

## Specializing for the repo

The personas in `council/` ship as generic skeletons. **They will produce generic critiques until specialized.** Edit each persona's `## Scope` section to describe:
- This repo's actual modules and their boundaries.
- The contracts other code or repos depend on.
- The security surface (auth model, trust boundaries, secret handling).
- The cost drivers (LLM calls, external services, storage).

Same for `scripts/security_checklist.md` — generic items at the top, repo-specific items added below them.
