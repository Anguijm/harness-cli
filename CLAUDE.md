# Claude Code Operating Protocol тАФ harness-cli

This file loads automatically at the start of every Claude Code session. Rules here are durable context the model should follow without re-explanation.

## Vocabulary

This repo follows the **harness engineering** discipline: the AI model is the engine, the surrounding systems are the harness. The harness has four kinds of components тАФ recognize them and treat them according to their role:

- **Guides (feedforward controls)** тАФ direct behavior *before* it happens. In this repo: this `CLAUDE.md`, persona files in `.harness/council/`, `.harness/scripts/security_checklist.md`, `.claude/settings.json`, `.claude/skills/`, `harness.yml`.
- **Sensors (feedback controls)** тАФ observe behavior and gate or alert *after* it happens. In this repo: gitleaks, lint, typecheck, unit tests (computational sensors); the Gemini council (inferential sensor); branch-guard.yml (post-hoc detector).
- **Topology** тАФ the canonical layout that ties guides + sensors together. Generated by `harness init`; verified by `harness check`. If the topology has drifted from the canonical, fix it before doing significant work.
- **Steering loop** тАФ the protocol of continuously refining guides and sensors based on what they actually catch and miss. Two artifacts: `.harness/failures.jsonl` (structured per-failure signal тАФ fields per the close-session skill) and `.harness/learnings.md` (human-readable narrative; KEEP / IMPROVE / INSIGHT / COUNCIL blocks).

The slogan: **structure in, structure out.** Free-form intent produces hallucinated file paths and invented APIs. Always ground a plan in the actual codebase before writing it (see Repository Impact Map below).

## Council = the inferential sensor that gates merge

Every change to `main` goes through a PR. `.github/workflows/council.yml` runs the multi-persona Gemini council on every PR open + push. The Lead Architect synthesis posts a single re-editable PR comment with a verdict: ЁЯЯв CLEAR | ЁЯЯб CONDITIONAL | ЁЯФ┤ BLOCK.

**Direct pushes to `main` are flagged by `.github/workflows/branch-guard.yml`** тАФ a post-hoc detector that fails on any push to `main` whose head commit is not associated with a merged PR. Detection, not prevention (GitHub Pro required for hard-block).

**Default: do not merge without ЁЯЯв.** CONDITIONAL = address remediations in a follow-up commit; council auto-reruns. BLOCK = rethink the change.

`[skip council]` in the PR title bypasses the workflow entirely; reserved for emergency hotfixes and trivial mechanical changes.

## Plan-first workflow with Repository Impact Map and Recall

For any non-trivial change:

1. **Recall past lessons first.** Run `harness recall "<theme of what you're about to build>"` to surface relevant entries from `.harness/learnings.md`, `.harness/failures.jsonl`, and prior council reports. Cheap, prevents you from re-learning things this repo already learned.
2. **Repository Impact Map.** Run `harness map "<feature description>"` (or do the equivalent by hand: grep for relevant symbols, list the actual files and tests likely to change). Write the result to `.harness/active_plan.md` as a "Repository Impact" block at the top.
3. **Write the plan** under the impact map in `.harness/active_plan.md`. Commit it (the council refuses to run against an untracked plan file).
4. **Run the local council** if you want pre-PR review: `python3 .harness/scripts/council.py --plan .harness/active_plan.md`.
5. **Implement** тАФ smallest diff that achieves the plan.
6. **Push and let CI run** тАФ both computational sensors (CI lint/typecheck/test) and the inferential sensor (council). Address remediations as follow-up commits.
7. **Merge after ЁЯЯв** + green CI.

## Pull requests тАФ do not auto-create

**Do NOT create a pull request unless the user explicitly asks for one.** `mcp__github__create_pull_request` is in `.claude/settings.json` under `permissions.ask`, so Claude Code prompts every time.

- "Commit and push" тЙа "open a PR". Push to the remote and stop.
- If a PR already exists for the branch, don't open a duplicate.

## Watch PRs automatically

**As soon as a PR is opened (by you or by the user), call `mcp__github__subscribe_pr_activity` to listen for PR events.** Don't wait to be asked. The council takes ~3 minutes to post its verdict; codex review comments arrive on a similar timeline; CI failures arrive as soon as a check completes. Subscribing is the right way to react in time without polling.

When events arrive:
- **Council comment (`<!-- council-report -->`)** тАФ read the verdict. CLEAR = nothing to do. CONDITIONAL = address remediations as follow-up commits. BLOCK = pause and discuss with the user before proceeding.
- **Codex review comments** тАФ sometimes GitHub adds Codex review comments alongside the council. Treat them as another sensor; respond if the comment is actionable, or explain in a reply if it's not.
- **CI failures** тАФ read the log, fix or surface the issue. Don't ignore.
- **User comments** тАФ defer to user direction.

If the user closes the session before events arrive, that's fine тАФ the next session can re-subscribe.

## Steering loop discipline

When a sensor catches something that the guides should have prevented (a council BLOCK that recurs, a CI failure pattern that repeats, a class of bugs the council keeps missing), the guide is wrong, not the sensor. Update the guide:

- Persona prompt missing context тЖТ edit the persona's `## Scope` section.
- Doctrine missing тЖТ add a section to this file.
- Anti-pattern needs to be banned тЖТ add to `.harness/scripts/security_checklist.md` or the relevant persona's non-negotiables.

The steering loop has **two artifacts** with different purposes:

- **`.harness/failures.jsonl` тАФ structured signal layer.** One JSONL line per recorded failure. Schema enforced by the `close-session` skill: `failure_class`, `sensor_involved`, `guide_gap`, `fix_sha`, `canonical_backport`. Programmatically queryable via `harness recall`. Compounds across sessions and repos. **This is the load-bearing record.**
- **`.harness/learnings.md` тАФ narrative layer.** Append-only KEEP / IMPROVE / INSIGHT / COUNCIL blocks per session. Human-readable; gives context and reasoning that a JSONL field can't carry. Pair every failures.jsonl entry with a one-line narrative here.

Never edit prior entries in either file; both are append-only.

**Before planning anything new, run `harness recall "<topic>"`** to see what past failures and learnings on that surface have already been recorded. Cheap, prevents re-learning.

## Backport rule тАФ fixes to harness files flow upstream

This repo's harness was generated from [`anguijm/harness-cli`](https://github.com/Anguijm/harness-cli). When you fix a bug or improve a file under `.harness/`, `.claude/`, `.github/workflows/`, `.husky/`, or `.gitleaks.toml`, **the fix has two homes:**

1. **This repo** тАФ land it as a normal PR.
2. **harness-cli** тАФ propose the same change against the canonical template, so other repos benefit when they upgrade their pinned `HARNESS_VERSION`.

If you don't backport, every other repo will hit the same bug independently. The downstream repos pinned to a canonical version (`HARNESS_VERSION` in `.github/workflows/drift-check.yml`) won't see the fix until you cut a new harness-cli release tag and they bump.

**What's NOT backported:** repo-specific customizations. A persona's `## Scope` section tailored to this repo, the security checklist's repo-specific entries, custom guard steps in council.yml (e.g., a grep that bans a specific anti-pattern in this repo's code) тАФ those stay here. The line is: "would another repo benefit from this exact change?" If yes, backport.

The `close-session` skill checks for harness-file edits at session end and prompts you about backporting before closing.

## Communication style

Default to plain language. Tell the user what you're doing in normal words ("committing your changes", "checking the council verdict", "pushing"), not in code or file paths. Surface technical detail (diffs, commands, file lists) only when the user asks or when a decision genuinely depends on it.

## Closeout

When the user says "close the session", "wrap up", or similar, run the `close-session` skill (`.claude/skills/close-session.md`). It commits, pushes, refreshes the handoff doc, appends to learnings, and verifies the branch is clean so the next session can pick up cold.

## Git hygiene (enforced by `.claude/hooks/check-branch-not-merged.sh`)

Before any `git push`, the harness fetches `origin/main` and refuses the push if the current branch has no content-difference from `origin/main` (the signature of a squash-merged branch). `git commit` is intentionally NOT intercepted.

If the hook fires: create a fresh branch from `origin/main`:

```bash
git fetch origin main
git checkout -b <new-branch> origin/main
```

## Halt protocol

If automation is producing bad output: `echo "reason" > .harness_halt` at repo root. The council and any cron workflows will silent-exit. See `.harness/halt_instructions.md`.
